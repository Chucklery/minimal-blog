# 输入作品链接自动解析图片/视频资源方案

## 目标

让用户不再需要打开 Chrome DevTools 手动复制网络资源地址。

用户只需要在软件里输入一个网页或作品链接，例如抖音某个用户的作品链接，软件自动识别平台、打开页面、发现里面的图片/视频资源，生成资源预览列表，最后由用户选择要下载哪些资源。

## 当前问题

现在项目主要能解析特定图片直链，例如：

- jpg
- png
- webp
- mp4

这类方式本质上是“下载器”，不是“资源发现器”。

它要求用户先自己找到真实资源地址，再交给软件解析。实际用户不会愿意这样操作，也不应该让用户依赖 DevTools。

## 推荐方向

把项目升级为：

输入链接 → 平台识别 → 插件解析 → 自动发现资源 → 资源预览 → 用户选择 → 下载

核心能力从“解析资源直链”升级为“解析页面里的资源”。

## 总体架构

用户输入 URL

↓

UrlRouter 识别平台

↓

PluginResolver 选择插件

↓

插件执行资源发现

↓

ResourceRanker 过滤和排序

↓

Flutter 前端展示预览

↓

用户勾选资源

↓

后端下载到本地

## 插件分层

### 1. direct-media 插件

用途：处理用户输入的图片、视频、音频直链。

识别类型：

- .jpg
- .jpeg
- .png
- .webp
- .avif
- .gif
- .mp4
- .m3u8
- .mp3

优点：

- 简单
- 快速
- 保留现有能力

缺点：

- 不能发现网页内部资源

这个插件继续保留，但不能作为主能力。

### 2. generic-html 插件

用途：处理普通静态网页。

后端用 HTTP 请求页面 HTML，然后解析：

- img src
- img srcset
- video src
- source src
- picture source
- meta og:image
- meta og:video
- link rel=preload
- script 中可能存在的图片/视频 URL

适合：

- 普通博客
- 图文页面
- 静态图片站
- 一些简单网页

不适合：

- 抖音
- 小红书
- 微博
- 需要 JS 渲染的动态网站

原因：很多动态网站的资源不是直接写在 HTML 里，而是 JS 执行后通过网络请求加载。

### 3. browser-discovery 插件

这是最关键的插件。

用途：后端自动启动浏览器，模拟用户打开网页，并监听页面加载过程中的网络资源。

它相当于软件自动做了用户原本在 Chrome DevTools Network 里做的事情。

流程：

1. 后端启动 Playwright / Chromium
2. 打开用户输入的作品链接
3. 等待页面加载
4. 监听所有 network request / response
5. 扫描 DOM 中的图片和视频节点
6. 滚动页面触发懒加载
7. 收集图片、视频、音频、封面等资源
8. 去重
9. 过滤无关资源
10. 返回资源预览列表

能收集的资源包括：

- image/jpeg
- image/png
- image/webp
- image/avif
- video/mp4
- application/vnd.apple.mpegurl
- audio/mpeg
- JSON 响应中的资源 URL

适合：

- 抖音
- 小红书
- 微博
- Twitter/X
- Instagram 类动态站点
- 其他 JS 动态加载页面

## 抖音插件设计

### 识别规则

DouyinPlugin.match(url) 应识别：

- douyin.com
- www.douyin.com
- v.douyin.com
- iesdouyin.com

### 解析流程

1. 接收用户输入的抖音链接
2. 如果是 v.douyin.com 短链接，先解析跳转到真实作品页
3. 启动 browser-discovery
4. 打开真实作品页
5. 监听网络请求
6. 扫描 DOM
7. 收集候选资源
8. 按抖音资源特征过滤和排序
9. 返回预览资源列表

### 重点收集对象

- 作品图集图片
- 视频播放地址
- 视频封面
- 高清图片
- og:image
- og:video
- douyinpic 域名资源
- douyinvod 域名资源

### 需要过滤的对象

抖音页面里会有很多无关资源，不能无脑全部展示。

需要过滤或降权：

- 头像
- icon
- logo
- svg
- 表情图
- tracking pixel
- 广告图
- 推荐流缩略图
- base64 图片
- 小尺寸图片
- 重复 CDN 地址

## ResourceRanker 评分规则

不能只把所有网络图片都列出来，否则用户看到的是一堆垃圾资源。

需要给每个资源打分。

### 高分资源

满足以下条件的资源应该排在前面：

- URL 包含 douyinpic
- URL 包含 douyinvod
- URL 包含 aweme
- content-type 是 image/jpeg
- content-type 是 image/webp
- content-type 是 video/mp4
- 尺寸大于 500x500
- 文件体积较大
- 来自 og:image
- 来自 og:video
- 来自页面主内容区

### 低分资源

满足以下条件的资源应该排在后面，或者默认隐藏：

- 小于 100x100
- URL 包含 avatar
- URL 包含 icon
- URL 包含 logo
- URL 包含 emoji
- URL 包含 sprite
- svg
- data:image
- tracking
- 广告域名
- 重复资源

### 前端展示策略

默认展示：

- score >= 阈值的资源
- 主图
- 视频
- 封面

提供开关：

- 显示全部资源
- 只看图片
- 只看视频
- 只看高清资源
- 隐藏疑似无关资源

## 后端 API 设计

### POST /api/preview

用途：输入链接，返回资源预览。

请求：

```json
{
  "url": "https://www.douyin.com/video/xxxx"
}
```

返回：

```json
{
  "platform": "douyin",
  "plugin": "douyin-browser-discovery",
  "source_url": "https://www.douyin.com/video/xxxx",
  "resources": [
    {
      "id": "res_001",
      "media_type": "image",
      "url": "https://example.com/image.webp",
      "preview_url": "https://example.com/image.webp",
      "filename": "douyin_作品ID_001.webp",
      "extension": "webp",
      "content_type": "image/webp",
      "size_bytes": 123456,
      "width": 1080,
      "height": 1920,
      "score": 95,
      "reason": "douyinpic large image"
    }
  ],
  "warnings": [],
  "errors": []
}
```

### POST /api/download

用途：用户选择资源后下载。

请求：

```json
{
  "resources": [
    {
      "id": "res_001",
      "url": "https://example.com/image.webp",
      "filename": "douyin_作品ID_001.webp"
    }
  ]
}
```

返回：

```json
{
  "downloaded_files": [
    "D:/workspace/ai-workspace/downloads/douyin/2026-06-06/douyin_作品ID_001.webp"
  ],
  "errors": []
}
```

## 技术方案选择

### 方案 A：Rust 直接控制浏览器

Rust 后端使用 chromiumoxide 或 headless_chrome 控制 Chromium。

优点：

- 后端语言统一
- 不需要 Node 子进程

缺点：

- CDP 细节多
- 开发难度高
- 调试成本高
- 监听 Network 和处理动态页面更麻烦

不建议作为第一阶段方案。

### 方案 B：Rust 调用 Node Playwright 脚本

Rust 后端收到链接后，调用 Node 脚本：

scripts/browser_probe.js

Node 脚本使用 Playwright 打开页面、监听网络、扫描 DOM，最后把资源 JSON 输出给 Rust。

优点：

- 实现最快
- Playwright 成熟稳定
- 浏览器自动化逻辑用 JS 写更自然
- 容易调试
- 适合快速做出产品

缺点：

- 项目多一个 Node 依赖
- 打包时要处理 Playwright 浏览器依赖

推荐作为当前项目的主方案。

### 方案 C：Chrome 扩展

用户在 Chrome 里打开抖音作品页，然后点击扩展按钮“解析当前页面”。

扩展读取当前页面资源，并发送给本地后端。

优点：

- 使用用户自己的浏览器环境
- 如果用户已登录抖音，插件能看到用户能访问的页面
- 比无头浏览器更不容易被风控
- 长期稳定性更好

缺点：

- 用户需要安装扩展
- 不是纯粹在 Flutter 软件中输入链接
- 扩展权限和安装流程需要设计

建议作为长期增强方案。

## 推荐实施路线

### 第一阶段：保留 direct-media

目标：保留当前直链解析能力。

改动：

- 不删除 direct-media 插件
- 让它继续处理图片、视频、音频直链
- 作为最快 fallback

验收：

- 输入 jpg 直链可以预览
- 输入 mp4 直链可以预览
- 用户可以选择下载

### 第二阶段：增强 generic-html

目标：普通网页输入 URL 后能自动发现 HTML 里的图片/视频。

需要实现：

- HTTP 请求网页
- 解析 HTML
- 提取 img/video/source/meta
- 提取 srcset
- URL 相对路径转绝对路径
- 去重
- 返回资源列表

验收：

- 输入普通网页地址，可以看到页面中的图片资源
- 不需要用户复制图片直链

### 第三阶段：新增 browser-discovery

目标：解决动态网页资源发现。

需要实现：

- 安装 Node Playwright
- 创建 scripts/browser_probe.js
- 后端调用该脚本
- 脚本打开页面
- 监听 Network
- 扫描 DOM
- 滚动页面触发懒加载
- 输出 JSON 给 Rust

验收：

- 输入动态页面 URL，可以发现页面加载过程中的图片和视频资源
- 用户无需打开 DevTools

### 第四阶段：新增 douyin 插件

目标：专门处理抖音作品链接。

需要实现：

- 识别 douyin.com 和 v.douyin.com
- 处理短链跳转
- 调用 browser-discovery
- 应用抖音专属过滤规则
- 提取作品 ID
- 生成更好的文件名
- 按作品目录保存

验收：

- 输入抖音作品链接，可以看到候选图片/视频资源
- 主资源排在前面
- 头像、icon、logo 不默认展示

### 第五阶段：前端体验优化

目标：让用户真正能用。

需要实现：

- 解析进度
- 资源卡片
- 图片缩略图
- 视频标记
- 全选/反选
- 类型筛选
- 显示全部资源开关
- 下载进度
- 错误提示

验收：

- 用户能清楚看到发现了哪些资源
- 用户能决定下载哪些资源
- 下载成功后能看到保存路径

### 第六阶段：可选 Chrome 扩展

目标：解决登录态、验证码、风控导致无头浏览器解析失败的问题。

需要实现：

- Chrome 扩展读取当前页面 DOM 资源
- 监听 performance.getEntriesByType('resource')
- 收集图片/视频资源
- POST 到本地 API
- Flutter 展示资源预览

验收：

- 用户在 Chrome 打开抖音页面
- 点击扩展按钮
- 本地软件收到资源列表
- 用户选择下载

## 合规和边界

需要明确：

1. 不做验证码绕过
2. 不做登录绕过
3. 不破解 DRM
4. 不绕过平台权限控制
5. 不做高频批量爬取
6. 请求需要限速
7. 只下载用户明确选择的资源
8. 对失败情况给出清楚提示

如果页面需要登录或验证码，软件应该提示：

“当前页面需要登录或验证。建议使用 Chrome 扩展方式从你自己的浏览器页面解析。”

## 最终推荐

短期采用：

Rust 后端 + Node Playwright browser_probe.js + Flutter 资源预览界面

长期增强：

增加 Chrome 扩展，用用户自己的浏览器环境解析当前页面资源。

这样可以同时满足：

- 用户只输入作品链接
- 自动识别平台
- 自动发现图片/视频资源
- 用户预览后选择下载
- 不需要打开 DevTools
- 保持插件化架构，后续可以扩展小红书、微博、Twitter/X 等平台
