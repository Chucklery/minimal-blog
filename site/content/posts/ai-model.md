# AI 时代开发者思维模型：从前端新手到全栈工程师

> 这份文档不是技术清单，而是你的“开发思考操作系统”。技术会变，但这些模型不会很快过时。

---

# 1. 先纠正一个误区

AI 盛行后，开发者不是“不需要基础”了，而是更需要判断力。

以前的问题是：

```txt
我不会写代码，怎么办？
```

现在的问题是：

```txt
AI 写了一堆代码，我怎么知道它对不对？
这个方案该不该接受？
系统以后会不会崩？
哪里该抽象？哪里不该抽象？
```

所以你真正要练的是：

1. 建模能力：把现实需求抽象成数据、流程、接口。
2. 拆解能力：把大任务拆成小任务。
3. 判断能力：知道方案好坏、代价、风险。
4. 验证能力：用测试、日志、运行结果证明代码有效。
5. 表达能力：能把系统讲清楚。

# 2. 开发者的五层模型

任何项目都可以拆成五层：

```txt
业务层：用户到底要完成什么？
产品层：页面和流程怎么设计？
前端层：UI、状态、交互怎么实现？
后端层：接口、业务规则、权限怎么实现？
数据层：数据如何存储、查询、关联？
```

你写代码前，先问：

```txt
我现在改的是哪一层？
这个改动会影响哪几层？
```

示例：新增“邀请成员”功能。

```txt
业务层：项目 owner 可以邀请用户加入。
产品层：项目设置页增加邀请弹窗。
前端层：表单、成员列表、权限选择。
后端层：邀请接口、权限校验。
数据层：project_members 表增加记录。
```

这就是全栈思维。

# 3. 需求分析模型

拿到需求，不要马上写代码。按这个模板问：

## 3.1 用户是谁？

```txt
谁在使用这个功能？
owner / editor / viewer / 游客？
```

## 3.2 用户要完成什么动作？

```txt
创建？查看？修改？删除？邀请？导出？生成？
```

## 3.3 成功结果是什么？

```txt
页面显示什么？
数据库新增什么？
接口返回什么？
用户下一步能做什么？
```

## 3.4 失败情况有哪些？

```txt
没权限怎么办？
数据不存在怎么办？
网络失败怎么办？
AI 返回格式错了怎么办？
重复提交怎么办？
```

## 3.5 边界是什么？

```txt
最多多少条数据？
是否分页？
是否需要权限？
是否需要日志？
是否影响已有功能？
```

# 4. 数据建模思维

资深开发者非常重视数据模型，因为数据模型错了，后面 UI 和接口都会别扭。

## 4.1 先找实体

实体通常是名词：

```txt
User
Project
Task
ProjectMember
Invitation
```

## 4.2 再找关系

```txt
User 可以拥有多个 Project
Project 可以有多个 Task
Project 可以有多个 Member
Member 属于一个 User 和一个 Project
```

## 4.3 再找行为

```txt
User 创建 Project
Project 创建 Task
Owner 邀请 Member
Editor 修改 Task
Viewer 查看 Task
```

## 4.4 JS 对象到数据库表

JS 对象：

```js
const task = {
  id: 't1',
  projectId: 'p1',
  title: '设计首页',
  status: 'todo'
}
```

数据库表：

```txt
tasks
- id
- project_id
- title
- status
- created_at
- updated_at
```

你可以把数据库表理解为“很多同类型 JS 对象的长期存储”。

# 5. 接口设计思维

接口是前端和后端的契约。

## 5.1 REST 资源模型

```txt
GET    /projects       获取项目列表
POST   /projects       创建项目
GET    /projects/:id   获取项目详情
PUT    /projects/:id   更新项目
DELETE /projects/:id   删除项目
```

不要这样设计：

```txt
/getProjectList
/createNewProject
/deleteProjectById
```

不是绝对不能用，而是不利于形成统一模型。

## 5.2 一个好接口要定义清楚

```txt
请求方法
路径
请求参数
请求 body
成功响应
错误响应
权限要求
边界情况
```

## 5.3 错误也是契约

```json
{
  "code": "NO_PERMISSION",
  "message": "你没有权限执行此操作"
}
```

前端看到 `NO_PERMISSION` 才知道应该提示用户、跳转、还是隐藏按钮。

# 6. 前端状态思维

前端最核心的问题：状态如何变成 UI。

```txt
Data -> State -> View -> User Action -> State Change -> View Update
```

## 6.1 三种状态

```txt
服务端状态：从 API 来的数据，比如 projects。
本地 UI 状态：弹窗开关、输入框内容、tab 当前项。
URL 状态：当前页面、详情 id、搜索 query。
```

## 6.2 不同状态放不同地方


| 状态         | 放哪里                   | 示例        |
| ------------ | ------------------------ | ----------- |
| 服务端数据   | React Query / SWR / hook | 项目列表    |
| 表单输入     | useState / form library  | 项目名称    |
| 页面参数     | URL / router             | projectId   |
| 全局用户信息 | store / context          | currentUser |

## 6.3 判断标准

问自己：

```txt
刷新页面后这个状态要不要保留？
多个页面要不要共享？
它是不是来自后端？
它能不能从已有数据推导出来？
```

能推导出来的状态，不要重复存。

# 7. 分层架构模型

一个清晰的前端项目可以这样分：

```txt
api       只负责请求
hooks     负责数据获取和状态逻辑
model     类型和业务模型
components  展示组件
pages     页面组合
utils     通用工具
```

后端可以这样分：

```txt
routes/controller   HTTP 输入输出
service             业务规则
repository/orm      数据访问
database            数据存储
```

核心原则：

```txt
外层依赖内层，内层不要依赖外层。
UI 可以调用 hook，hook 可以调用 api。
api 不应该知道 UI 长什么样。
```

# 8. 抽象能力训练

抽象不是把代码写复杂，而是找到稳定边界。

## 8.1 什么时候该抽象？

可以抽象：

* 同样逻辑出现 3 次以上。
* 业务概念稳定，比如 Project、Task、Member。
* 需要被测试。
* 需要被多个页面复用。

不要急着抽象：

* 需求还不稳定。
* 只是两行代码重复。
* 你还没理解变化方向。
* 抽象后名字说不清楚。

## 8.2 好抽象的标准

```txt
名字清楚
职责单一
输入输出明确
隐藏复杂度
不泄露实现细节
```

## 8.3 每天做一个抽象练习

看一个功能，写下：

```txt
这个功能的核心实体是什么？
核心动作是什么？
哪些逻辑可能复用？
哪些东西未来可能变化？
如果变化，影响范围在哪里？
```

# 9. 资深开发者的取舍思维

技术选型不是问“哪个最好”，而是问“哪个适合当前约束”。

## 9.1 取舍维度

```txt
开发速度
学习成本
团队熟悉度
生态成熟度
性能
可维护性
部署复杂度
长期扩展
```

## 9.2 示例：Express vs NestJS

Express：

* 优点：简单、自由、上手快。
* 缺点：项目大了容易乱，需要自己定结构。

NestJS：

* 优点：结构清晰，适合大型项目。
* 缺点：概念多，新手成本高。

结论：

* 一个月学习项目：Express 更快。
* 想训练企业级架构：NestJS 更好。

这就是取舍，不是站队。

# 10. AI 协作开发模型

AI 的正确位置：

```txt
你：定义问题、拆任务、定边界、审查结果。
AI：生成方案、补代码、解释概念、写测试、找问题。
```

错误用法：

```txt
帮我做一个完整项目。
```

正确用法：

```txt
我正在做任务管理系统。
现在要实现项目列表接口。
已有数据模型如下：...
要求：
1. GET /api/projects 支持分页。
2. 只能返回当前用户有权限的项目。
3. 错误格式统一为 { code, message }。
4. 请先给设计方案，不要直接写代码。
```

## 10.1 AI 任务模板

```txt
角色：你是资深全栈工程师。
背景：我在做 xxx 项目，技术栈是 xxx。
目标：实现 xxx 功能。
现状：已有文件结构 / 代码如下。
约束：不要改动 xxx；保持 xxx 分层。
验收：运行 xxx 命令通过；页面表现为 xxx。
输出：先给方案，再列修改文件，最后给测试方式。
```

## 10.2 AI 代码审查模板

```txt
请审查下面代码，重点看：
1. 是否有安全问题。
2. 是否有状态管理问题。
3. 是否有边界情况遗漏。
4. 是否违反分层结构。
5. 是否需要测试。
请按 严重 / 中等 / 建议 分类输出。
```

## 10.3 AI 输出验收清单

```txt
代码能运行吗？
测试通过吗？
是否改了不该改的文件？
是否引入新依赖？
错误状态处理了吗？
权限校验了吗？
类型是否正确？
命名是否清楚？
```

# 11. 调试思维

不要靠猜。按顺序查。

## 11.1 四步法

```txt
1. 复现：稳定复现问题。
2. 定位：确定是前端、接口、后端、数据库哪一层。
3. 缩小：找到最小出错代码。
4. 修复：改完后验证，不要只看表面。
```

## 11.2 前后端问题定位

```txt
页面没显示数据：
1. DevTools 看请求发出去了吗？
2. Network 看接口返回了吗？
3. 返回结构和前端预期一致吗？
4. React state 更新了吗？
5. JSX 渲染条件对吗？
```

## 11.3 后端问题定位

```txt
接口报错：
1. 路由是否匹配？
2. 参数是否正确？
3. controller 收到请求了吗？
4. service 逻辑是否正确？
5. 数据库查询是否正确？
6. 错误是否被统一处理？
```

# 12. 从新手到资深的路线

## 阶段 1：会写

特征：

* 能跟教程写页面。
* 能实现简单功能。
* 主要靠搜索和 AI。

重点：

* JS/TS 基础。
* React 基础。
* 基本 API 调用。

## 阶段 2：会拆

特征：

* 能拆组件。
* 能拆 API 层。
* 能拆 hook。
* 知道哪些逻辑不该放页面里。

重点：

* 分层。
* 命名。
* 状态管理。

## 阶段 3：会联调

特征：

* 能做完整 CRUD。
* 能处理前后端契约。
* 能处理错误、权限、loading。

重点：

* REST API。
* 数据库。
* 认证授权。

## 阶段 4：会设计

特征：

* 写代码前会画结构。
* 能解释为什么这样设计。
* 能预测哪里会变、哪里会坏。

重点：

* 数据建模。
* 系统设计。
* 取舍分析。

## 阶段 5：会带项目

特征：

* 能拆任务给别人或 AI。
* 能审查代码。
* 能制定规范。
* 能为质量负责。

重点：

* 工程规范。
* 测试。
* 文档。
* 交付。

# 14. 技术问题三层回答法

任何技术问题都用这个结构回答。

```txt
表层：它是什么。
中层：它解决什么问题，怎么用。
深层：为什么这样设计，有什么取舍，什么坑。
```

示例：什么是 REST API？

表层：
REST API 是一种基于 HTTP 资源的接口设计风格。

中层：
它用 GET、POST、PUT、DELETE 表示对资源的查询、创建、更新、删除，比如 `/projects/:id` 表示某个项目资源。

深层：
它的价值是让接口语义统一、前后端契约清晰。但复杂业务动作不一定都适合硬套 REST，有时需要 RPC 风格接口，比如 `/projects/:id/invite`。

# 15. 最重要的学习方式

## 15.1 反向学习

不要从概念开始死记。你适合从真实文件反推。

流程：

```txt
看一个真实功能文件
找到不懂的 API
判断：必须理解 / 只需认识
把代码改一点
运行验证
写一句自己的解释
```

## 15.2 项目驱动

所有知识都绑定到项目：

```txt
学 useEffect -> 做项目列表请求
学 REST -> 做项目 CRUD
学数据库 -> 存项目和任务
学权限 -> owner/editor/viewer
学 AI -> 任务自动拆解
学部署 -> 项目上线
```

## 15.3 AI 辅助但不外包思考

你可以让 AI：

* 解释代码。
* 生成草稿。
* 写测试。
* 找 bug。
* 比较方案。

但你必须自己做：

* 判断需求。
* 设计边界。
* 决定取舍。
* 运行验证。
* 接受责任。

# 16. 一句话总结

AI 时代，普通开发者拼的是“会不会调用 AI”；优秀开发者拼的是“能不能定义问题、拆解系统、审查结果、持续交付”。

你的目标不是背更多 API，而是建立这个闭环：

```txt
需求理解 -> 系统建模 -> 任务拆解 -> AI 协作 -> 编码实现 -> 测试验证 -> 文档复盘 -> 下一轮迭代
```
