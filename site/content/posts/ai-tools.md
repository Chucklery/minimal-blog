# AI 工具高效开发指南

> 这份文档解决一个核心问题：如何把 AI 当成开发搭档，而不是把项目交给 AI 瞎写。

---

# 1. AI 开发的正确流程

推荐流程：

```txt
1. 你先定义需求
2. 你拆分任务
3. AI 给方案
4. 你审查方案
5. AI 写小块代码
6. 你运行验证
7. AI 辅助修 bug
8. 你写复盘和文档
```

不要这样：

```txt
帮我写一个完整系统。
```

这会导致：

* 文件结构混乱。
* 代码你看不懂。
* 隐藏 bug 很多。
* 后续无法维护。
* 项目不像你的项目。

---

# 2. 给 AI 任务的五要素

每次让 AI 做开发任务，必须给这 5 个东西：

## 2.1 背景

```txt
我在做一个 AI 项目任务管理系统。
技术栈：React + TypeScript + Node.js + PostgreSQL。
当前已有项目列表和任务列表。
```

## 2.2 目标

```txt
我要新增“项目成员权限”功能。
```

## 2.3 现状

```txt
已有表：users、projects、tasks。
已有接口：GET /projects、POST /projects。
```

## 2.4 约束

```txt
不要直接在 React 组件里写 fetch。
保持 api -> hooks -> components -> pages 分层。
后端保持 controller -> service -> database 分层。
```

## 2.5 验收标准

```txt
owner 可以邀请成员。
editor 可以创建任务。
viewer 只能查看。
无权限时接口返回 403。
前端显示错误提示。
```

# 3. 通用提示词模板

## 3.1 方案设计模板

```txt
你是资深全栈工程师。

背景：
我正在做 xxx 项目，技术栈是 xxx。

目标：
实现 xxx 功能。

现有结构：
- 前端：api / hooks / components / pages
- 后端：controller / service / database

约束：
1. 不要直接写代码。
2. 先分析需求和边界。
3. 给出数据模型、接口设计、前端改动点、后端改动点。
4. 说明可能风险。

输出格式：
1. 需求理解
2. 数据模型
3. API 设计
4. 前端改动
5. 后端改动
6. 测试方式
7. 风险与取舍
```

## 3.2 写代码模板

```txt
基于上面的方案，请只实现第 N 步。

要求：
1. 只修改以下文件：xxx。
2. 不要改动无关文件。
3. 保持 TypeScript 类型明确。
4. 错误处理使用统一格式。
5. 写完后告诉我如何运行测试。

请输出：
- 修改后的代码
- 修改原因
- 测试方法
```

## 3.3 解释代码模板

```txt
请用“JS 新手能懂”的方式解释下面代码。
要求：
1. 先说这段代码整体做什么。
2. 再逐行解释。
3. 把 TypeScript 类型映射到普通 JS 理解。
4. 标出我必须掌握的 API 和只需认识的 API。
5. 最后给一个更简单的 JS 类比。
```

## 3.4 Debug 模板

```txt
我遇到 bug：xxx。

现象：xxx。
期望：xxx。
实际：xxx。
相关代码：xxx。
报错信息：xxx。
我已经尝试：xxx。

请按以下顺序分析：
1. 可能原因列表。
2. 最可能的 1-2 个原因。
3. 如何验证。
4. 最小修复方案。
5. 修复后如何测试。
```

## 3.5 代码审查模板

```txt
请审查下面代码。
重点看：
1. 逻辑 bug。
2. 类型问题。
3. 权限漏洞。
4. 错误处理。
5. 状态管理。
6. 是否违反分层。
7. 是否有可维护性问题。

输出格式：
- 严重问题
- 中等问题
- 建议优化
- 不建议修改的部分
```

# 4. AI 不适合直接做什么

AI 不适合直接决定：

```txt
产品边界
业务规则
技术选型最终决策
安全策略
数据库核心模型
是否上线
是否删除重要文件
```

这些必须你自己判断，AI 只能辅助分析。

---

# 5. AI 适合做什么

AI 很适合：

```txt
解释概念
生成代码草稿
补全类型
写单元测试
生成 mock 数据
重构小函数
找明显 bug
生成文档初稿
比较技术方案
```

最适合 AI 的任务有两个特点：

1. 边界清楚。
2. 可验证。

例如：

```txt
请给 ProjectService 的 createProject 写单元测试。
```

比这个好得多：

```txt
帮我优化后端。
```

# 6. 小步快跑原则

每次只让 AI 做一个小任务。

错误：

```txt
帮我实现登录、权限、项目管理、任务管理、AI 拆任务和部署。
```

正确：

```txt
第一步：请设计 users 表和登录接口。
第二步：请实现 POST /auth/login。
第三步：请写登录接口测试。
第四步：请实现前端登录表单。
```

为什么：

* 小任务容易检查。
* 小任务容易回滚。
* 小任务更不容易跑偏。
* 小任务更适合学习。

# 7. AI 输出验收流程

AI 写完代码后，你必须做 6 件事：

```txt
1. 看文件：它改了哪些文件？
2. 看结构：有没有破坏分层？
3. 看逻辑：是否满足需求？
4. 看边界：错误、空值、权限处理了吗？
5. 跑项目：能不能启动？
6. 跑测试：有没有真实验证？
```

如果没有测试，至少做手动验收：

```txt
正常输入
空输入
错误输入
无权限用户
重复提交
网络失败
刷新页面
```

# 8. AI 生成代码常见问题

## 8.1 编造不存在的 API

AI 可能会写：

```js
router.getCurrentUser()
```

但你的项目里根本没有这个函数。

处理方式：

* 让 AI 先读取真实文件。
* 要求它只使用现有 API。
* 你自己搜索确认。

## 8.2 过度抽象

AI 喜欢创建很多 helper、manager、factory。

处理方式：

* 一个月学习项目优先简单直接。
* 重复 3 次以上再抽象。
* 抽象名字说不清楚就不要抽。

## 8.3 忽略错误状态

AI 常只写成功路径。

你要追问：

```txt
如果接口失败怎么办？
如果用户没权限怎么办？
如果数据为空怎么办？
如果 AI 返回 JSON 格式不对怎么办？
```

## 8.4 类型看起来对，运行不对

TypeScript 只能检查一部分问题。

必须运行：

```txt
npm run build
npm run test
手动打开页面验证
```

# 9. 学习时如何用 AI

## 9.1 不懂代码时

提示词：

```txt
请把下面 React/TS 代码翻译成普通 JS 思维。
我重点想知道：
1. 数据从哪里来。
2. 状态在哪里变化。
3. UI 如何根据状态变化。
4. 哪些 TS 类型必须理解。
```

## 9.2 学新 API 时

提示词：

```txt
请解释 xxx API。
要求：
1. 它解决什么问题。
2. 不用它会怎样。
3. 最小示例。
4. 在真实项目里的使用场景。
5. 常见坑。
```

## 9.3 学架构时

提示词：

```txt
请比较 api/hooks/components/pages 这种前端分层。
要求：
1. 每层职责。
2. 数据流方向。
3. 错误示例。
4. 正确示例。
5. 如果项目变大如何扩展。
```

# 10. 一个月项目中的 AI 使用安排


| 阶段    | AI 用法                    | 你必须做的事           |
| ------- | -------------------------- | ---------------------- |
| 第 1 周 | 解释 React/TS、生成小组件  | 自己理解数据流和状态   |
| 第 2 周 | 辅助设计 API、生成后端样板 | 自己审查数据模型和权限 |
| 第 3 周 | 辅助联调、写测试、查 bug   | 自己运行验证和修边界   |
| 第 4 周 | 辅助写文档、做代码审查     | 自己做最终取舍和复盘   |

# 11. 最小项目验收清单

你的项目至少要满足：

## 前端

* [ ]  有登录页。
* [ ]  有项目列表。
* [ ]  有项目详情。
* [ ]  有任务 CRUD。
* [ ]  有 loading / error / empty。
* [ ]  有基础权限展示。

## 后端

* [ ]  有用户认证。
* [ ]  有项目 API。
* [ ]  有任务 API。
* [ ]  有权限校验。
* [ ]  有统一错误格式。
* [ ]  有数据库持久化。

## AI

* [ ]  有一个明确 AI 功能。
* [ ]  AI 输出格式被校验。
* [ ]  AI 失败时有 fallback。
* [ ]  不把 API key 暴露到前端。

## 工程

* [ ]  有 README。
* [ ]  有 API 文档。
* [ ]  有数据库设计文档。
* [ ]  有基本测试或手动验收记录。
* [ ]  能部署。

# 12. 推荐工作流

每天开发时按这个流程：

```txt
1. 写今天目标：一句话。
2. 写验收标准：3-5 条。
3. 让 AI 给方案，不写代码。
4. 你审查方案。
5. 让 AI 实现一个小步骤。
6. 你运行。
7. 报错就让 AI 基于真实报错分析。
8. 通过后写复盘。
```

示例：

```txt
今日目标：实现任务创建功能。
验收标准：
1. 用户可以输入任务标题。
2. 点击创建后调用 POST /tasks。
3. 创建成功后列表刷新。
4. 创建失败时显示错误。
5. 空标题不能提交。
```

# 13. 你要避免的学习方式

不要这样学：

```txt
今天看 React 课程 5 小时。
明天看 Node 课程 5 小时。
后天看数据库课程 5 小时。
```

这叫输入很多，产出很少。

应该这样学：

```txt
我今天要让项目列表从后端真实读取。
不会的地方再查 React useEffect、fetch、接口设计。
最后必须跑起来。
```

项目驱动，缺什么学什么。

---

# 14. 最终原则

AI 可以让你写得更快，但不能替你负责。

真正高效的学习方式是：

```txt
用项目制造问题；
用 AI 加速解决；
用测试验证结果；
用复盘沉淀模型。
```

只要坚持这个闭环，一个月后你不会变成“真正资深”，但你会明显脱离“只会跟教程写页面”的阶段，进入“能独立拆解和交付项目”的轨道。
